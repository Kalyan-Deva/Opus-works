## About the book

Code is read far more often than it is written, yet most of us were trained only to make it work. *Legible Code* is about the other ninety percent of a program's life: the months and years during which other people, including your future self, have to understand it, change it, and trust it.

This is not a style guide, and it is not a list of rules to enforce in a linter. It is a book about judgment. Why does one function feel obvious while another, doing the same thing, feels like a puzzle? Why do some codebases invite contribution and others repel it? The answers live in small, repeated decisions: how you name things, where you place a boundary, how much you choose to say and how much you leave out.

The book argues that legibility is an act of respect for the next reader, and that this respect compounds into faster teams, fewer defects, and software that ages gracefully instead of rotting.

## What's inside

- Naming as design: how good names reveal intent and bad ones hide it
- Functions and modules that are easy to hold in your head
- Structuring code so the important parts are obvious and the rest stays out of the way
- Comments that earn their place, and the ones that should have been code
- Managing complexity instead of merely relocating it
- Reviewing for readability without turning review into a battlefield
- Refactoring toward clarity in a codebase you did not write

## Who it's for

This book is for working programmers at every level who care about the craft, not just the output. Newer engineers will find a vocabulary for instincts they are still forming. Experienced engineers will find language for the things they already do well and want to teach others.

It is language-agnostic by design. The examples are deliberately small and the principles apply whether you write `Python`, `Go`, `TypeScript`, or something the author has never heard of. If you have shipped code and then had to read it again later, you already understand the problem this book is trying to solve.

By the end, you should see your own work through a reader's eyes, and find that writing legible code is not slower. It is simply better.

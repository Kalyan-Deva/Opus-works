A leaked API key in a git repository is one of the most common and most damaging mistakes in software. This post covers why it happens and the layered controls that prevent it.

## Why an `.env` in git is bad

Committing secrets feels convenient, but git never forgets. Even if you delete the file in a later commit, the value lives forever in history and in every clone and fork. Anyone with read access, now or in the future, can recover it. Worse, public repositories are scraped continuously, and exposed cloud keys are often abused within minutes.

The fix is to keep plaintext secrets out of version control entirely and source them from a dedicated system at runtime.

## Catch leaks before they land

Scan for secrets in CI and as a local pre-commit hook so a credential never reaches the remote. Gitleaks is a widely used open-source scanner.

```bash title="scan.sh" showLineNumbers
# Scan the working tree
gitleaks detect --source . --verbose

# Scan staged changes in a pre-commit hook
gitleaks protect --staged --redact
```

If a secret does leak, scanning is only the first step. You must also **rotate** the exposed credential immediately, because it must be considered compromised regardless of how quickly you scrub history.

## Store secrets in a vault

Centralized secret managers give you encryption at rest, access control, and audit logging. Common choices include HashiCorp Vault and the cloud-native managers such as AWS Secrets Manager, Azure Key Vault, and Google Secret Manager.

```bash title="vault.sh" showLineNumbers
# Write a secret
vault kv put secret/app/db password="$(openssl rand -base64 24)"

# Read it back at runtime
vault kv get -field=password secret/app/db
```

## Encrypt files you must keep in git

Sometimes you genuinely want configuration in the repo, for example for GitOps. SOPS encrypts only the values, leaving keys readable so diffs stay meaningful. It integrates with cloud KMS or age keys.

```bash title="sops.sh" showLineNumbers
# Encrypt in place using an age recipient
sops --encrypt --age age1ql3z... secrets.yaml > secrets.enc.yaml

# Decrypt at deploy time
sops --decrypt secrets.enc.yaml
```

## Prefer short-lived, dynamic credentials

The most robust approach is to avoid long-lived secrets altogether. Vault can generate dynamic database credentials that expire automatically. Cloud providers support OIDC federation so a CI job exchanges a short-lived token for temporary access without any stored key.

> Tip: A credential that lives for five minutes cannot be abused for long even if it leaks. Short TTLs turn a catastrophic exposure into a minor one.

## Rotate on a schedule

Even well-protected static secrets should rotate. Build rotation into the lifecycle:

1. Generate the new secret in the vault.
2. Roll it out to consumers, supporting both old and new briefly.
3. Revoke the old value and confirm nothing breaks.

## Summary

| Control | Purpose |
| --- | --- |
| Gitleaks | Stop secrets reaching the remote |
| Vault / cloud managers | Central storage with audit and access control |
| SOPS | Safely keep encrypted config in git |
| OIDC / dynamic creds | Eliminate long-lived secrets |

Layer these together and the question stops being "did we leak a secret" and becomes "how quickly does an exposed secret expire."

Modern applications are mostly other people's code. A typical service pulls in hundreds of transitive dependencies, and your build pipeline is itself an attack surface. Supply chain security is about answering two questions with evidence: *what is in my software*, and *can I trust how it was built*.

## Know what you ship: SBOMs

A **Software Bill of Materials (SBOM)** is a machine-readable inventory of every component and version in an artifact. Two open formats dominate:

- **CycloneDX** — security-focused, widely used for vulnerability workflows.
- **SPDX** — an ISO-standardized format, strong on licensing and provenance.

With an SBOM you can answer "am I affected?" the moment a vulnerability in a dependency is disclosed, instead of grepping through builds by hand.

Generate one in CI:

```bash title="sbom.sh" showLineNumbers {2}
# Generate a CycloneDX SBOM from the project directory
syft dir:. -o cyclonedx-json > sbom.cdx.json

# Scan that SBOM for known vulnerabilities
grype sbom:sbom.cdx.json --fail-on high
```

## Trust how it was built: provenance and SLSA

**SLSA** (Supply-chain Levels for Software Artifacts) is a framework describing increasing levels of build integrity. The key idea is **provenance**: a signed, tamper-evident record of how an artifact was produced, by which builder, from which source.

The levels build on each other:

- **Build L1** — provenance exists and is available.
- **Build L2** — provenance is signed and generated by a hosted build service.
- **Build L3** — the build runs on hardened, isolated infrastructure that prevents provenance forgery.

Higher levels make it progressively harder for an attacker to inject malicious build steps without detection.

## Sign your artifacts: Sigstore and cosign

Provenance and SBOMs are only useful if consumers can verify they are authentic. **Sigstore** provides keyless signing using short-lived certificates tied to a workload identity, so you do not manage long-lived private keys. Sign a container image in CI with `cosign`:

```yaml title=".github/workflows/release.yml" showLineNumbers {9,11}
permissions:
  id-token: write      # required for keyless OIDC signing
  packages: write
jobs:
  sign:
    runs-on: ubuntu-latest
    steps:
      - name: Sign image
        run: cosign sign --yes ghcr.io/acme/api@${DIGEST}
      - name: Attach SBOM as attestation
        run: cosign attest --yes --predicate sbom.cdx.json \
             --type cyclonedx ghcr.io/acme/api@${DIGEST}
```

Note that signing references the image **digest** (`@sha256:...`), not a mutable tag.

## Pin everything

Floating versions are how a compromised release sneaks in. Pin dependencies and base images to exact, immutable identifiers:

- Commit lockfiles (`package-lock.json`, `poetry.lock`, `go.sum`) and verify hashes.
- Reference container images by digest, not `:latest`.
- Pin GitHub Actions to a full commit SHA rather than a tag.

> Tip: Verification is only real if you enforce it. Add an admission policy that rejects unsigned images, so an unsigned artifact cannot reach production even by accident.

## Putting it together

A defensible pipeline produces an SBOM, scans it, generates SLSA provenance, signs the artifact and its attestations with cosign, and refuses to deploy anything that fails verification. None of these steps is exotic anymore. Adopt them incrementally, starting with SBOM generation, and you turn "trust me" into "here is the evidence."

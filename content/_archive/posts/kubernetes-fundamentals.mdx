Kubernetes has a large API, but you can be productive with a handful of objects and one big idea: declarative reconciliation. Everything else builds on that foundation.

## The declarative model

You do not tell Kubernetes *how* to do things. You declare the desired state of the world in YAML, and controllers continuously work to make reality match. Each controller runs a loop: observe the current state, compare it to the desired state, and take action to close the gap. If a node dies and takes your Pods with it, the controller notices the mismatch and schedules replacements. You never issued a "restart" command; you only declared "I want three replicas."

## Core objects

- **Pod:** the smallest deployable unit, one or more containers that share a network namespace and storage. You rarely create Pods directly.
- **Deployment:** manages a ReplicaSet to keep a desired number of Pod copies running, and handles rolling updates and rollbacks.
- **Service:** a stable virtual IP and DNS name that load-balances across a changing set of Pods.
- **ConfigMap / Secret:** inject non-sensitive and sensitive configuration into Pods as env vars or mounted files.
- **Ingress:** routes external HTTP/HTTPS traffic to Services based on host and path rules.

## Namespaces

Namespaces partition a cluster into virtual sub-clusters for teams or environments. They scope names, resource quotas, and access policies, so two teams can both run a Service called `api` without collision.

## A minimal Deployment and Service

This manifest runs three replicas of an nginx container and exposes them through a Service. The highlighted lines are the parts that wire everything together.

```yaml title="app.yaml" showLineNumbers {8,15,20,27}
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web
spec:
  replicas: 3
  selector:
    matchLabels:
      app: web
  template:
    metadata:
      labels:
        app: web
    spec:
      containers:
        - name: web
          image: nginx:1.27
          ports:
            - containerPort: 80
---
apiVersion: v1
kind: Service
metadata:
  name: web
spec:
  selector:
    app: web
  ports:
    - port: 80
      targetPort: 80
```

The Service `selector` matches the Pod template `labels`. That label match is the glue: the Deployment stamps `app: web` onto its Pods, and the Service routes to any Pod carrying that label, no matter how often Pods come and go.

## Applying and observing

```bash title="deploy.sh" showLineNumbers {1,3}
kubectl apply -f app.yaml
kubectl get pods -l app=web
kubectl rollout status deployment/web
```

`kubectl apply` sends your desired state to the API server; the controllers do the rest.

> Tip: Set resource `requests` and `limits` on every container. Without requests, the scheduler cannot place Pods sensibly, and without limits a single Pod can starve its neighbors.

Master these objects and the reconciliation loop, and the rest of Kubernetes becomes variations on a theme you already understand.
